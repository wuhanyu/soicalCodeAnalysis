Comment 222 by cbent...@chromium.org, Apr 19, 2010 
lukemorey: Negotiate typically means Kerberos or NTLM under the hood (it negotiates 
the strongest authentication scheme which both client and server support, and uses 
that). 

If you are seeing a username/password prompt it is for one of two reasons:
  a) The site you are trying to access is not considered part of the local intranet 
security zone. Typically if you include a "." character in the host name, it will not 
be part of the local intranet zone unless configuration changes have been made.
  b) Integrated Authentication has been attempted once, but the server rejected it. 
In this case the user will be prompted. The best way to detect this is either to run 
a packet capture tool such as wireshark, or enable logging in Chrome. 

If you know what the SPN should be for the service that you are trying to 
authenticate to, that would also be helpful information. Essentially, Chrome (and 
other browsers) have some heuristics about generating the SPN based on the hostname 
in the URL, but it isn't guaranteed to match.

Comment 247 by francip, May 07, 2010 
On further investigation, it does not appear that problem is the presence of the WWW-
Authenticate: NTLM header. I run quick test where I modified the response and removed 
the NTLM header and left only the Negotiate. The sequence still failed. So it appears 
that the problem is somewhere in the negotiation sequence itself.

A quick comparison between an IE request and a Chrome request to the same site shows 
indeed that the first two request/response pairs after the Negotiate are quite 
similar (with very small differences in the tokens). However, after the negotiation, 
IE sends a third request that has an Authorization header with a token, to which IIS 
replies with HTTP 200. Chrome though sends the third request without the 
Authorization header, to which IIS replies again (and rightfully so) with HTTP 401 
and the WWW-Authenticate header.

So it looks like the SPNEGO/NTLM tokens exchange works (at least it looks right in 
Fiddler :-)), but Chrome "forgets" to send the Authorization header with the last 
request after the negotiation is finished.

Of course, there might be a good reason why Chrome wouldn't send the Authorization 
header with the last request. However, it would be interesting to understand why it 
happens.
