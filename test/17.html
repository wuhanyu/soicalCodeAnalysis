This isn't a bug.  "type=search" is not a valid attribute, so the standards-compliant
behavior is to drop it, resulting in an <input> tag which is equivalent to <input
type="text">.

Comment 9 by ojan@chromium.org, Sep 08, 2008 
Chrome team is not opposed to implementing input type=search. In fact, we would like 
to be as close as possible to WebKit rendering. We really don't want to give web 
developers another rendering engine to have to worry about.

There was some plumbing necessary to make it work and, given the low usage of this 
feature, we deemed it not a launch blocker. In order to get this implemented, we 
would need a UI for the element that matches Windows style form elements (i.e. no 
rounded corners). I'm not sure what else needs to be done, but patches are welcome.

Comment 11 by maciej, Sep 25, 2008 
There are many WebKit features which are not yet standards, as is true for any other browser. I do not think 
either WebKit generally or Chrome's version of WebKit in particular should remove nonstandard features, 
particularly ones that are aimed at standards track. <input type=search> has been proposed for HTML5 and the 
editor has not yet made a final decision.
Comment 12 by ojan@chromium.org, Sep 25, 2008 
Just to be 100% clear, the only reason we don't have this implemented right now is a
question of priorities, *not* a question of standards. We intend to implement this as
soon as someone finds some time to do so (or even better, someone so moved sends us a
patch).

When I say priorities, I mean as compared to other compatibility bugs that exist in
Chromium vs. Webkit. There are other more severe bugs that we should get to first.

gothicx, to your point, HTML5 attempts to codify what browsers do. So, once a number
of browsers implement a given feature, it is made part of the HTML5 spec, assuming
it's not totally crazy. Stated differently, the browsers drive the standard just as
much as the standard drives the browsers.
